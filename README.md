# BusinessSlideVQA Evaluator

フロンティアVLM（Claude Sonnet 4.6 / GPT-4.1 / Qwen2.5-VL-72B）の日本語ビジネスドキュメント読解性能を比較評価する Gradio Web アプリです。

[BusinessSlideVQA](https://github.com/stockmarkteam/business-slide-questions)（ストックマーク社）のベンチマークを用い、日本語ビジネススライド理解における VLM の性能感と失敗傾向を探索的に確認することを目的としています。

本プロジェクトの関心は、単なるモデル順位の比較ではなく、**どの種類のスライドで、どの種類の誤りが起きやすいのか** を観察し、今後の改善論点を切り出すことにあります。

---

## 評価の位置づけ

Stockmark 社の技術ブログ (https://stockmark-tech.hatenablog.com/entry/2025/06/03/101007) では、図表やテキストが混在するドキュメント理解を評価するためのデータセットとして `BusinessSlideVQA` が紹介されています。  
それを踏まえ、本プロジェクトでは、外部から利用可能な高性能フロンティアVLMを使って、日本語ビジネススライド理解でどの程度の性能感が出るのかをまず確認しました。

現段階では、厳密なベンチマーク構築というより、**性能感の把握と失敗モードの予備的観察** を目的とした評価プロトタイプとして位置づけています。

---

## 評価方法

- 評価対象モデル：
  - Claude Sonnet 4.6
  - GPT-4.1
  - Qwen2.5-VL-72B
- データ：
  - BusinessSlideVQA から画像タイプ別に均等サンプリングした30問
- 判定方法：
  - Claude Sonnet 4.6 を LLM-as-a-Judge として使用し、各モデルの回答を 1〜5 のスコアで評価

### 注意点
- 評価サンプルは30問であり、統計的な一般化を意図したものではありません
- Judge として Claude Sonnet 4.6 を用いているため、評価バイアスの可能性があります
- 以下の考察は、探索的評価に基づく予備的な観察です

## 主な観察結果

- 今回の探索的評価では、**Claude Sonnet 4.6** が総合平均で最も高かった
- 総合平均だけでなく、**モデルごとに崩れやすい画像タイプが異なる** 点が重要だった
- **円グラフ** は全モデル満点で、今回のサンプルでは比較的容易なタスクだった可能性が高い
- **画像** カテゴリは全モデル 4.60 で同点であり、今回のサンプルではモデル差が出にくかった可能性がある
- グラフ系でも、円グラフ・棒グラフ・積み上げ棒グラフ・折れ線グラフで挙動が異なり、**種類ごとに必要能力が異なる可能性** が示唆された

### モデル別の予備的考察

- **Claude Sonnet 4.6**
  - 全体として安定して高スコア
  - テキスト・棒グラフ・積み上げ棒グラフで強い
  - 一方で折れ線グラフなどでは取りこぼしもあり、万能と断定する段階ではない

- **GPT-4.1**
  - 総合平均は Qwen2.5-VL-72B と同等
  - 表や画像では一定の性能を示す
  - 棒グラフ・折れ線グラフで相対的に低く、グラフ読解で不安定さがある可能性

- **Qwen2.5-VL-72B**
  - 棒グラフ・折れ線グラフは良好
  - 一方で表・積み上げ棒グラフ・テキストで相対的に低い
  - 単純なグラフ読解には比較的強く、複雑な構造理解で弱さが出る可能性

### 画像タイプ別の予備的考察

- **棒グラフ**
  - GPT-4.1 が相対的に低スコア
  - 数値読み取りや比較関係の把握に課題がある可能性

- **積み上げ棒グラフ**
  - Qwen2.5-VL-72B が相対的に低スコア
  - 系列分解や構造理解の難しさが影響している可能性

- **表**
  - Qwen2.5-VL-72B が相対的に低スコア
  - 表構造理解やセル対応の難しさが示唆される

- **折れ線グラフ**
  - 今回のサンプルでは Qwen2.5-VL-72B が相対的に良好
  - ただし、一般化にはより大きなサンプルが必要

## この結果から言えること / まだ言えないこと

### 言えること
- カテゴリごとの得意不得意の差が見えてきた
- 日本語ビジネススライド理解では、表・グラフ・構造表現が識別力のある評価軸になっている可能性がある
- 「グラフ理解」を一括りにせず、グラフ種類ごとに分けて見る意義がある

### まだ言えないこと
- どのモデルが一般に最も優れているか
- 低スコアの原因が、数値読み取り・構造理解・文脈理解のどれなのか
- 今回の傾向が、より大規模なサンプルでも再現するか

---

## 今後の改善方針

- 個票レベルで回答を確認し、失敗モードを分類する
- OCR依存の失敗、構造理解の失敗、数値読解の失敗、もっともらしい誤答を切り分ける
- 評価サンプル数を増やし、カテゴリ別傾向の安定性を確認する
- LLM-as-a-Judge の妥当性を検証し、Judge の複線化も検討する
- 実務上クリティカルな誤りに重みをつけた評価設計を検討する

## セットアップ

```bash
# 1. リポジトリクローン
git clone <this-repo>
cd business-slide-vqa-eval

# 2. 仮想環境
python -m venv .venv
source .venv/bin/activate

# 3. 依存パッケージ
pip install -r requirements.txt

# 4. APIキー設定
cp .env.example .env
# .env を編集してAPIキーを設定

# 5. データ準備（BusinessSlideVQAリポジトリが ../business-slide-questions/ にある前提）
python prepare_data.py

# 6. 起動
python app.py
```

## HuggingFace Spaces デプロイ

1. HF Spaces で新しい Gradio Space を作成
2. コードをプッシュ
3. Settings > Secrets に `ANTHROPIC_API_KEY`, `OPENAI_API_KEY`, `OPENROUTER_API_KEY` を設定
4. `data/images/` は HF Dataset としてアップロードするか、初回起動時にダウンロードする仕組みに変更
